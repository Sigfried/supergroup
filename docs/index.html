supergroup

Grouping for flat records that returns navigable structure. Live docs — every code cell below is editable; press Run. Everything in cell scope is also on window for devtools experimentation.

What it is

supergroup groups flat record collections — CSV rows, query results, event logs — into navigable structure. Where d3.group returns nested Maps (grouping, but no navigation) and d3.hierarchy navigates already-shaped trees (navigation, but no records and no multi-parent), supergroup fills the seam: every node carries its records, parents, children, path, depth, and aggregates. It also builds DAGs from parent-id data, sequence trees from event successors, and comparisons between two groupings. Zero dependencies; adapters emit plain objects for d3 and dag-browser-widget. In a d3 callback holding a datum that might have come from anywhere, the node carries its records, path, and ancestors — everything a tooltip needs, without re-deriving it.

The data

Datasets used on this page, all static files under data/ and all attached to window once loaded:

  • athletes — Olympic athletes by country/sport/year (public Olympics dataset).
  • patients — a small synthetic hospital-visit CSV (fabricated, not real patient data).
  • budgets — a two-year department budget table, made up for the compare examples.
  • hurricanes — HURDAT2 Atlantic hurricane best-track status events.
  • fips — U.S. state/county FIPS codes.
  • containment — a small hand-written specimen/study parent-id taxonomy, for DAG examples.
  • conditions, drugs, persons — a synthea 1k-patient synthetic-patient OMOP extract (public S3), joined to OMOP vocabulary (SNOMED/RxNorm/ATC concept names).
  • drugClasses — the ATC + RxNorm classification closure over the drugs this cohort used, built from the same OMOP vocabulary; fromParentIds-ready. See data/curation/README.md for how it's built.

Quick start / core

Start with flat records — rows from a CSV file, a SQL query, an AJAX call. An array of plain objects renders as a table (first 10 rows):

// some very fake hospital data, loaded from a CSV file
toTable(patients, { maxRows: 10 })

// group by country, sport, year — every level keeps its records
sg = supergroup(athletes, ['Country', 'Sport', 'Year'])
summary(sg)

// same collection, shown as a tree (depth- and breadth-limited)
prettyPrint(sg, { maxDepth: 2, maxChildren: 8 })

// navigation: node lookup by path, paths, parents
sg = supergroup(athletes, ['Country', 'Sport', 'Year'])
n = sg.node(['United States', 'Swimming'])
;({
  label: n.label, dim: n.dim, depth: n.depth,
  namePath: n.namePath(), records: n.records.length,
  children: n.children.map(String),
})

// aggregates: agg over an accessor; union-safe rollup
sg = supergroup(athletes, ['Country', 'Sport'])
us = sg.node(['United States'])
;({
  medals: us.agg(r => r['Total Medals']),
  rollup: us.rollup(r => r['Total Medals']),
  pctOfAllRecords: us.pct().toFixed(3),
})

The same hospital-visit CSV that appears throughout this codebase's history: grouping by physician and unit, then rolling up charges.

// hospital visits grouped by physician, then unit
sg = supergroup(patients, ['Physician', 'Unit'])
adams = sg.node(['Adams'])
;({
  physicians: sg.roots.map(String),
  adamsUnits: adams.children.map(String),
  adamsRecords: adams.records.length,
  adamsCharge: adams.rollup(r => r.Charge),
})

// lookup + namePath on a deeper path
sg = supergroup(patients, ['Physician', 'Unit', 'Patient'])
n = sg.node(['Adams', 'preop'])
;({ namePath: n.namePath(), patients: n.children.map(String), records: n.records.length })

Multi-valued dims split a record into more than one group without duplicating it in the record set; the Insurance column here is semicolon-delimited (a patient can carry more than one plan).

// multi-valued dims (Insurance: "Aetna PPO;Medicare") and Date keys
sg = supergroup(patients, [
  { by: r => r.Insurance.split(';'), name: 'Insurance', multi: true },
  'Date',
])
;({ plans: sg.roots.map(String), recordsTotal: sg.nodes.reduce((n, x) => n + (x.children.length ? 0 : x.records.length), 0) })

d3.group returns nested Maps: fast, but each node is just a key and a value — no path, no parent, no aggregate. The same grouping via supergroup keeps records at every level and adds navigation.

// same grouping, two shapes: d3.group vs supergroup
grouped = d3.group(athletes, r => r.Country, r => r.Sport)
sg = supergroup(athletes, ['Country', 'Sport'])
;({
  d3group: [...grouped.get('United States').keys()].slice(0, 5),
  supergroup: sg.node(['United States']).children.map(String).slice(0, 5),
  notice: 'the Map has no records, no parent, no path — just nested keys',
})

DAG module + interactive browser

Parent-id data (a table with an id and one or more parent ids per row) builds a DAG: nodes can have multiple parents, and cycles are detected and reported as backedges rather than causing infinite recursion.

// parent-id data → DAG; multi-parent nodes appear once, cycles become backedges
g = fromParentIds(containment)
;({
  roots: g.roots.map(String),
  multiParent: g.nodes.filter(n => n.parents.length > 1).map(String),
  backedges: g.backedges.map(e => `${e.parent} ⟲ ${e.child}`),
})

// records meet the DAG: attach by node id, then union-safe rollups
g = fromParentIds(containment, {
  records: [
    { entity: 'Specimen', n: 12 }, { entity: 'Aliquot', n: 30 },
    { entity: 'Assay', n: 7 }, { entity: 'Response', n: 55 },
  ],
  recordKey: r => r.entity,
})
visit = g.nodes.find(n => n.id === 'Visit')
// union-then-sum: shared descendants counted once, never per-path
;({ under: visit.rollup(r => r.n), induced: subgraph(g, ['Visit', 'Specimen', 'Aliquot']).nodes.map(String) })

fromParentChild builds the same kind of DAG straight from a flat parent/child row table — no pre-shaped id list required.

// fromParentChild: a taxonomy expressed as plain parent/child rows
rows = [
  { parent: 'Animal', child: 'Vertebrate' },
  { parent: 'Animal', child: 'Invertebrate' },
  { parent: 'Vertebrate', child: 'Mammal' },
  { parent: 'Vertebrate', child: 'Bird' },
]
g = fromParentChild(rows, { parent: 'parent', child: 'child' })
;({ roots: g.roots.map(String), nodes: g.nodes.map(String) })

Sequence trees

Given records that link to their successor (and, optionally, predecessor), groupBySequence groups by a key at each step and builds a tree of what-happens-next (or what-happened-before, or both).

// link each storm's events into next/prev chains; keep each storm's first event
byStorm = d3.group(hurricanes, r => r.hur_id)
for (const evts of byStorm.values()) {
  evts.sort((a, b) => d3.ascending(a.date, b.date))
  evts.forEach((e, i) => { e.next = evts[i + 1] ?? null; e.prev = evts[i - 1] ?? null })
}
hurricaneStarts = [...byStorm.values()].map(e => e[0])
;`${byStorm.size} storms linked`

// what sequences of statuses do storms move through?
groupBySequence(hurricaneStarts, { key: 'status', next: e => e.next, direction: 'forward', maxDepth: 3 })

// anchored-both: align every storm on its first Hurricane event, look both ways
anchors = hurricaneStarts
  .map(e => { let x = e; while (x && x.status !== 'Hurricane') x = x.next; return x })
  .filter(Boolean)
groupBySequence(anchors, { key: 'status', next: e => e.next, prev: e => e.prev, direction: 'both', maxDepth: 2 })

// toD3 + d3.partition: sequence tree as an icicle
seq = groupBySequence(hurricaneStarts, { key: 'status', next: e => e.next, direction: 'forward', maxDepth: 4 })
const root = d3.hierarchy(toD3(seq)).sum(d => (d.children ? 0 : d.records.length))
d3.partition().size([320, 640])(root)
const color = d3.scaleOrdinal(d3.schemeTableau10)
const svg = d3.create('svg').attr('width', 640).attr('height', 320)
const cell = svg.selectAll('g').data(root.descendants()).join('g')
  .attr('transform', d => `translate(${d.y0},${d.x0})`)
cell.append('rect')
  .attr('width', d => d.y1 - d.y0 - 1).attr('height', d => Math.max(0, d.x1 - d.x0 - 1))
  .attr('fill', d => color(d.data.name))
cell.filter(d => d.x1 - d.x0 > 14).append('text')
  .attr('x', 4).attr('y', 13).attr('font-size', 11)
  .text(d => `${d.data.name} (${d.value})`)
svg.node()

The most-migrated v1 idiom, asRootVal(), wrapped a flat list in a single synthetic root so it could feed a d3 hierarchy layout. In v2 that's a {root: 'synthetic'} option plus toD3:

// v1's asRootVal(), in v2: a synthetic root wrapping a flat grouping
sg = supergroup(athletes, 'Sport', { root: 'synthetic' })
toD3(sg)

Real sequence data: condition histories

Real-world OMOP data contains unmapped source records (concept_id 0, "No matching concept"); the linking cell below filters those out before building sequences.

// link each patient's conditions into a chain; keep each patient's first diagnosis
mapped = conditions.filter(r => r.concept_id !== 0)
byPerson = d3.group(mapped, r => r.person_id)
for (const evts of byPerson.values()) {
  evts.sort((a, b) => d3.ascending(a.start_date, b.start_date))
  evts.forEach((e, i) => { e.next = evts[i + 1] ?? null; e.prev = evts[i - 1] ?? null })
}
firstConditions = [...byPerson.values()].map(e => e[0])
;`${byPerson.size} patients linked (${conditions.length - mapped.length} unmapped rows dropped)`

// what conditions tend to follow a patient's first diagnosis?
groupBySequence(firstConditions, { key: 'condition', next: e => e.next, direction: 'forward', maxDepth: 3 })

// anchored-both: align every patient on their first diagnosis of the most common condition
counts = d3.rollup(firstConditions, v => v.length, r => r.condition)
const [commonCondition] = [...counts.entries()].sort((a, b) => b[1] - a[1])[0]
anchors = firstConditions
  .map(e => { let x = e; while (x && x.condition !== commonCondition) x = x.next; return x })
  .filter(Boolean)
groupBySequence(anchors, { key: 'condition', next: e => e.next, prev: e => e.prev, direction: 'both', maxDepth: 2 })

Compare

compare takes two groupings over the same shape and returns a merged tree where every node carries cmp.in ('a', 'b', or 'both') and cmp.countDelta.

// same hierarchy, two years of budgets — where did counts move?
y2010 = supergroup(budgets.filter(r => r.Year === 2010), 'Dept')
y2011 = supergroup(budgets.filter(r => r.Year === 2011), 'Dept')
diffBudgets = compare(y2010, y2011)
prettyPrint(diffBudgets)

Each node's summary line shows cmp.in and, when nonzero, Δ = countDelta. Comparing two DAG attachments needs { by: 'id' } — a node's identity is its DAG id, not its tree path.

// cohort comparison over one classification: women vs men, by node id
gender = new Map(persons.map(p => [p.person_id, p.gender]))
cohort = g => fromParentIds(drugClasses,
  { records: drugs.filter(r => gender.get(r.person_id) === g), recordKey: r => String(r.concept_id) })
diff = compare(cohort('F'), cohort('M'), { by: 'id' })
interesting = diff.nodes.filter(n => n.cmp && Math.abs(n.cmp.countDelta) > 25)
interesting.map(n => `${n.label}: ${n.cmp.in} Δ${n.cmp.countDelta}`).join('\n')
