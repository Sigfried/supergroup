# DuckDB on the demo page — design

Date: 2026-07-14. Status: approved design, pre-plan.
Relates to: docs milestone Task 9 examples round (feedback bullet 6);
does not replace the rest of that round.

## Goal

Let visitors run SQL (duckdb-wasm, fully in-browser, serverless) in the
live-docs page and feed results into supergroup. Flagship use: derive the
ATC/RxNorm hierarchy live from real OMOP vocabulary tables — replacing the
precomputed `drug-classes.json` — join synthea drug records to it, and
browse the DAG with per-node record and distinct-patient counts.

SQL is positioned as the data-prep/join layer (what supergroup deliberately
lacks); supergroup remains the star: recursive union-then-aggregate rollups
over a multi-parent DAG are painful in SQL and one line in supergroup.

## Decisions made during brainstorming

- **No query server.** duckdb-wasm in the browser reads static parquet
  over HTTP range requests. Same serverless model as vs-hub.
- **No precomputed hierarchy files.** The page derives hierarchy from
  `concept` + direct-edge tables at runtime via SQL.
- **Data volume**: as much as feasible; page paints immediately, data may
  take a few seconds to become ready in the background.
- **Vocab cut**: ALL ATC + RxNorm concepts and all SNOMED condition-domain
  concepts, with `concept_ancestor` `min_levels_of_separation = 1` direct
  edges. No demo-drug closure filtering (Task 7b's two-pass closure hack
  becomes unnecessary — there is no closure boundary).
- **Synthea tables**: all ten tables from the public bucket, full columns.
  Paring moves from extract time to page-authoring time (a prior paring
  decision cost us `drug_era_end_date`; not repeating that).
- **Hosting**: a separate data repo with GitHub Pages (serves
  `Access-Control-Allow-Origin: *` and supports range requests). Reusable
  by vs-hub and future projects; keeps supergroup's git history lean. The
  `synthea-omop` S3 bucket is NOT CORS-enabled (verified) so the page
  cannot read it directly.
- **Format: parquet, not a `.duckdb` database file.** Parquet is readable
  by everything (vs-hub, Python, R, Arrow); a `.duckdb` file couples the
  repo to DuckDB storage-format versioning (duckdb-wasm lags CLI releases)
  and duplicates data. Query performance at this scale is a wash; "one
  database" ergonomics are identical since views unify the namespace.
- **Init strategy (approach A)**: background init after first paint, not
  lazy-on-first-query (cold start would make the flagship example feel
  broken) and not a notebook-framework migration (see below).
- **Cell model**: both a `sql()` global for JS-cell composition and a
  dedicated SQL cell flavor for readability. Stays on the bespoke
  livecells runtime: Observable Notebook Kit / Framework were evaluated
  and rejected because their built static sites are read-only for readers
  — "every cell is editable" is this page's identity. (Their standalone
  `@observablehq/inspector` may be adopted later for object display; out
  of scope here.)

## Part 1 — data repo

New repo `omop-demo-data` (name provisional), GitHub Pages enabled. Files
are committed artifacts regenerated by script; no build service.

```
omop-demo-data/
  build/extract.sh          # psql (db n3c, schema n3c) + duckdb → parquet
  vocab/
    concept.parquet         # concept_id, concept_name, vocabulary_id,
                            #   concept_class_id, standard_concept,
                            #   concept_code, domain_id
    concept_edge.parquet    # parent_id, child_id
                            #   (concept_ancestor @ min_sep = 1)
  synthea1k/                # all ten tables, all columns, CSV → parquet
    person.parquet          # 1,130 persons
    drug_era.parquet        # incl. end_date, exposure_count, gap_days
    condition_era.parquet
    drug_exposure.parquet   # incl. days_supply, quantity, visit ids
    condition_occurrence.parquet
    visit_occurrence.parquet
    procedure_occurrence.parquet
    measurement.parquet
    observation.parquet
    observation_period.parquet
  README.md                 # provenance; verbatim Synthea citation
                            #   (task-9-feedback bullet 5); vocabulary
                            #   snapshot date; licensing notes (SNOMED/
                            #   ATC/RxNorm publishing approved, Task 7b
                            #   amendment)
```

Vocab rows in scope: ATC ≈ 7.2k concepts, RxNorm ≈ 313k, SNOMED condition
domain ≈ 160k, plus their min-sep-1 edges. Estimated 30–50MB parquet
total; every file far under GitHub's 100MB limit.

Notes:

- `concept_id = 0` sentinel rows in era/occurrence/exposure tables are
  KEPT (true fact about real OMOP data; teaching material), documented in
  the README.
- The bucket has no `device_exposure` table — no device data exists in
  this dataset; `procedure_occurrence` is procedures only.
- Extraction: synthea CSVs via `curl` from
  `https://synthea-omop.s3.amazonaws.com/synthea1k/` (one loop, all
  tables); vocab via `psql` against local `n3c` then duckdb `COPY … TO
  … (FORMAT parquet)`.

## Part 2 — page-side runtime (`docs/duckdb.js`)

- **Init**: kicked off automatically after first paint. Loads duckdb-wasm
  from jsdelivr (`getJsDelivrBundles()` selects the wasm flavor); creates
  an in-memory database; `CREATE VIEW <name> AS SELECT * FROM
  read_parquet('<data-repo-url>/…')` for each remote parquet file;
  registers the page's locally fetched datasets (the existing `data`
  object) as DuckDB tables so SQL and JS cells see the same data. The
  data-repo base URL is a single const, overridable for local testing
  against a local checkout of the data repo.
- **`sql()` global**: `` sql(`SELECT …`) `` returns a thenable resolving
  to an array of PLAIN JS row objects. Arrow structures are unwrapped at
  the boundary: `BigInt` → `Number` (throw with a clear message if a value
  exceeds `Number.MAX_SAFE_INTEGER`), Arrow timestamps/dates → JS `Date`,
  everything else as-is. No Arrow types leak into cell scope, because
  supergroup, `toTable`, and the JSON renderer all expect plain records.
- **Readiness**: `sql()` awaits the init promise internally; queries
  issued during warm-up simply wait. A small status line near the page top
  shows `duckdb: loading… → ready (n tables)` / `failed: <reason>`.
- **Failure isolation**: if the CDN or data repo is unreachable, `sql()`
  rejects with a clear error (rendered in the cell's error style);
  JS-only cells are unaffected.

## Part 3 — cell mechanics (`docs/livecells.js`)

- **Await-before-publish (general runner change)**: today only the
  rendered result is awaited if thenable. Change: after a run, for each
  name the cell published, if `window[name]` is a thenable, await it and
  replace both `window[name]` and the publish ledger entry with the
  resolved value. Net effect: `rows = sql(\`…\`)` behaves as if top-level
  `await` existed (it doesn't, in eval'd code). Rendering of a published
  thenable result awaits the same promise (no double-execution).
- **SQL cell flavor**: `<pre class="cell sql" data-name="edges">…</pre>`.
  Same wrap/bar (Run, Clear, status, published-names line) as JS cells;
  CodeMirror uses `@codemirror/lang-sql`. Run passes the cell text to
  `sql()`; result rows render as an HTML table; rows publish to
  `window[data-name]` (ownership/Clear semantics identical to JS-cell
  publishes). No `data-name` → display-only. SQL cells never touch eval,
  so the ASI semicolon-prefix gotcha does not apply to them.
- **Table renderer unification (DRY)**: the dataset-preview `htmlTable`
  in livecells.js becomes the shared table renderer used by dataset cards
  and SQL results (row-count cap only on request, matching the
  formatting-module truncation philosophy).
- **JS cells**: `sql` is added to cell scope/window like any other global.

## Part 4 — library addition + flagship example

- **`rollup` distinct counts** (core, `src/node.ts`): extend
  `SGNode.rollup` to accept a distinct-key option, e.g.
  `node.rollup({ distinct: r => r.person_id })` → result gains
  `distinct: number` (count of unique key values over the
  union-of-node-and-descendants record set — union first, then distinct;
  never sum-over-paths). Exact signature reconciled with the existing
  `rollup(accessor?)` in the plan; covered by unit tests. Feeds the
  bullet-7 v1-friendliness inventory.
- **Flagship authored cells** (page content, order):
  1. SQL cell: `WITH RECURSIVE` over `concept_edge` climbing from the
     ingredient ids present in `drug_era` up through ATC; emits
     `parent_id, child_id, concept_name` for the reachable subgraph.
     Recursion to fixed point replaces Task 7b's two-pass closure.
  2. JS cell: `fromParentChild(edges, …)` → `attachRecords(atc, drugEra,
     r => String(r.drug_concept_id))` → DBW render with
     `label · N pts · M rx` baked into node names via
     `rollup({ distinct })`. (DBW has no count column — upstream issue;
     name-baking is the workaround and will hit DBW's known
     horizontal-clipping bug, feedback bullet 8.)
  3. SQL cell (cohort slicing): same records filtered by a demographic
     join (e.g. `gender = 'F' AND year_of_birth < 1960`), re-attach,
     re-render — the counts change. This is the exploration loop.
- Existing `drug-classes.json` + its `curate.sh` machinery are retired
  once the flagship cells replace their usage (removal timing decided in
  the plan; the dataset-card section changes belong to the look round).

## Error handling

- duckdb init failure → status line shows failure; `sql()` rejects; the
  rest of the page works.
- `sql()` query errors surface DuckDB's message in the cell error style.
- BigInt overflow on unwrap → explicit error naming the column.
- Unmatched records in `attachRecords` are already reported
  (`{matched, unmatched}`) — example cells display the match rate rather
  than hiding it.

## Testing

- Library (`rollup` distinct): vitest unit tests alongside the existing
  rollup tests (multi-parent dedup case included).
- Page: no JS test harness exists for livecells; verification follows the
  established pattern — `node --check`, `npm run typecheck`, headless
  Chrome `--dump-dom` against `?runall` plus targeted checks (duckdb
  status reaches ready; flagship cells produce `sg-leaf` nodes; zero
  `cell-error`). Headless run needs network (CDN + data repo) — noted in
  the plan so it isn't mistaken for flakiness.
- Data repo: extract script ends with row-count assertions (concept and
  edge counts per vocabulary printed and compared against psql counts).

## Non-goals / out of scope

- No notebook-framework migration; no reactive runtime.
- No DBW count-column feature (upstream issue to be filed, bullet 8).
- No `@observablehq/inspector` adoption (possible look-round item).
- No changes to the remaining examples-round bullets (5, 7 inventory
  follow-through, 8) beyond what this feature naturally touches.
- No synthea100k or additional OMOP tables beyond the bucket's ten
  (future lifeflow/timelines work).
